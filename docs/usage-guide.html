Recommended for operators

일반 사용자는 Portable GUI부터 시작하세요

프로그래밍이나 CLI 작업이 필요하지 않은 사용자는 Windows artifact의 WwiseOriginalsRelocator.exe를 사용합니다.

한 번만 압축 해제

실행 파일과 _internal 폴더를 함께 유지하면 Python 설치 없이 실행할 수 있습니다.

읽기 전용 GUI

환경 확인, 이동 계획, 사전 검증 사유와 보고서만 제공합니다. apply와 rollback은 노출하지 않습니다.

Portable 배포물 구성

WwiseOriginalsRelocator.exe, _internal, 사용가이드.html, LICENSE.txt, VERSION.txt가 함께 있어야 합니다.

화면 중심 사용법은 Portable GUI 오프라인 가이드를 확인하세요. 아래 절부터는 단일 파일 apply/rollback을 검증하는 개발·운영 담당자용 CLI 절차입니다.

CLI에서 먼저 알아둘 것

이 도구의 기본 단위는 검토가 끝난 WAV 한 파일입니다. 계획 단계까지는 읽기 전용이며, 실제 변경은 반드시 명시적인 apply --only와 롤백 manifest를 통해서만 진행합니다.

도구가 하는 일

WAAPI/WWU에서 소스를 찾고, 이동 계획을 만든 뒤 p4 move와 GUID 범위의 정확한 WWU 패치를 수행합니다.

도구가 하지 않는 일

배치 적용, changelist submit, 애매하거나 공유된 소스의 자동 이동은 수행하지 않습니다.

  1. doctor
    준비 상태 확인
  2. scan
    Wwise 객체 수집
  3. plan
    이동안 생성
  4. validate-plan
    사전 검증
  5. apply --only
    한 파일 적용
  6. Wwise 재시작
    외부 변경 로드
  7. validate-apply
    사후 검증
  8. rollback
    원상 복구
안전 계약

도구는 changelist를 submit하지 않습니다. 테스트 중에도 baseline 이외의 relocation changelist는 submit하지 마세요.

Setup

설치와 기본 테스트

저장소 루트에서 Python 3.11 이상을 사용합니다. WAAPI 기능까지 포함하려면 프로젝트 전용 가상환경에 optional dependency를 설치하세요.

macOS / Linux · 가상환경 및 개발 의존성
python3 -m venv .venv
.venv/bin/python -m pip install -e ".[dev,waapi]"
Windows PowerShell · 가상환경 및 개발 의존성
py -3.11 -m venv .venv
& .\.venv\Scripts\python.exe -m pip install -e ".[dev,waapi]"

자동화 테스트

macOS / Linux · 전체 단위 테스트
.venv/bin/python -m pytest
Windows PowerShell · 전체 단위 테스트
& .\.venv\Scripts\python.exe -m pytest

성공: 모든 테스트가 통과하고 종료 코드가 0입니다. 이 테스트는 parser, planner, preflight, apply/rollback, P4 오류 판별, WAAPI reader, pilot project 생성을 다룹니다.

CLI가 설치되었는지 확인

macOS / Linux · 도움말
.venv/bin/wwise-p4-source-relocator --help
Windows PowerShell · 도움말
& .\.venv\Scripts\python.exe -m wwise_p4_source_relocator --help

읽기 전용 점검부터 시작하기

Wwise나 Perforce 없이도 포함된 fixture WWU를 파싱하고, 변경 없는 inspection plan을 만들 수 있습니다.

macOS / Linux · Fixture WWU 검사
mkdir -p reports

.venv/bin/wwise-p4-source-relocator inspect-wwu \
  --wwu "tests/fixtures/sample_project/Actor-Mixer Hierarchy/Default Work Unit.wwu" \
  --project-root tests/fixtures/sample_project \
  --json-out reports/source-plan.json \
  --markdown-out reports/source-plan.md
예상 결과

발견한 source reference 수가 출력되고, 모든 항목은 의도적으로 skip 처리됩니다. 이 명령은 파일을 옮기거나 WWU를 수정하지 않습니다.

실제 프로젝트 준비 상태 확인

doctor는 apply 전에 반드시 통과해야 하는 첫 게이트입니다. .wproj 수, Originals WAV, WWU source reference, p4, workspace mapping, waapi-client, WAAPI 연결을 확인합니다.

Doctor — 읽기 전용
.venv/bin/wwise-p4-source-relocator doctor \
  --project-root "/absolute/path/to/WwiseProject" \
  --waapi-port 18080 \
  --json-out reports/pilot-readiness.json \
  --markdown-out reports/pilot-readiness.md

중단: 하나라도 실패하면 실제 적용으로 넘어가지 말고 보고서의 blocker를 먼저 해결합니다.

Wwise 테스트 프로젝트 만들기

bootstrap-project는 비어 있는 경로에만 새 Wwise 프로젝트를 생성합니다. 기존 .wproj는 건드리지 않습니다.

macOS · 테스트 프로젝트 생성
export PILOT_ROOT=/private/tmp/wwise-relocator-p4/workspace/WwiseRelocatorPilot

.venv/bin/wwise-p4-source-relocator bootstrap-project \
  --project-root "$PILOT_ROOT"

생성되는 fixture는 다음 불일치를 의도적으로 포함합니다.

항목생성 값계획 결과
Wwise 객체\Containers\Default Work Unit\VO\Script\CH04카테고리는 Script
현재 WAVOriginals/Voices/English(US)/Scenario/CH04/CH04_S102_WT_001.wav잘못된 Scenario 위치
예상 WAVOriginals/Voices/English(US)/Script/CH04/CH04_S102_WT_001.wavmove-and-patch

프로젝트 루트의 relocator-pilot.json에는 스캔 root와 예상 source/target 경로가 기록됩니다.

대상 경로 보호

destination이 비어 있지 않으면 bootstrap이 거부합니다. 테스트를 다시 만들 때는 새 디렉터리 이름을 사용하세요. Windows 명령은 Windows 통합 테스트의 5번 절차를 따릅니다.

.tools/perforce/p4d -r /tmp/wwise-relocator-p4/server '-cset dm.user.hideinvalid=0'

.tools/perforce/p4 -p 127.0.0.1:16661 -u pilot passwd
.tools/perforce/p4 -p 127.0.0.1:16661 -u pilot login

Client root 규칙

macOS에서는 /private/tmp 사용

Python은 /tmp를 /private/tmp로 해석합니다. P4 client의 Root도 /private/tmp/wwise-relocator-p4/workspace로 지정해야 workspace preflight가 일치합니다.

test project를 depot에 reconcile하고 fixture baseline만 submit합니다. relocation에는 별도의 numbered pending changelist를 만들되 submit하지 않습니다. 상세 client spec과 binary 검증 절차는 Local Perforce Pilot 문서를 참고하세요.

macOS · End-to-end

macOS: Wwise + Perforce 통합 테스트

아래 예시는 macOS Wwise 2025.1.8 경로, WAAPI WAMP 18080, HTTP 18090, pending changelist 2를 사용합니다. 설치 경로와 changelist 번호는 현재 환경에 맞게 바꾸세요.

1. WAAPI server 시작

별도 터미널 — 계속 실행
"/Applications/Audiokinetic/Wwise_2025.1.8.9170/Wwise.app/Contents/Tools/WwiseConsole.sh" \
  waapi-server \
  "$PILOT_ROOT/WwiseRelocatorPilot.wproj" \
  --wamp-port 18080 \
  --http-port 18090 \
  --allowed-origin localhost,127.0.0.1

2. 준비 상태 확인

Doctor
mkdir -p reports

.venv/bin/wwise-p4-source-relocator doctor \
  --project-root "$PILOT_ROOT" \
  --waapi-port 18080 \
  --json-out reports/live-readiness.json \
  --markdown-out reports/live-readiness.md

3. Scan → plan → preflight

변경 전 계획
.venv/bin/wwise-p4-source-relocator scan \
  --project-root "$PILOT_ROOT" \
  --object-root "\Containers\Default Work Unit\VO" \
  --chapter CH04 \
  --waapi-url ws://127.0.0.1:18080/waapi \
  --out reports/live-scan.json

.venv/bin/wwise-p4-source-relocator plan \
  --scan reports/live-scan.json \
  --out reports/live-plan.json

.venv/bin/wwise-p4-source-relocator validate-plan \
  --plan reports/live-plan.json \
  --report reports/live-preflight.md
검토 게이트

reports/live-plan.json과 자동 생성되는 Markdown을 열어 source, target, object GUID, Work Unit이 의도와 일치하는지 사람이 확인한 뒤에만 진행합니다.

4. 한 파일 적용

Single-file apply
.venv/bin/wwise-p4-source-relocator apply \
  --plan reports/live-plan.json \
  --only CH04_S102_WT_001.wav \
  --changelist 2 \
  --manifest reports/live-manifest.json

manifest는 첫 mutating P4 명령보다 먼저 저장됩니다. WWU patch 또는 로컬 검증이 실패하면 도구가 manifest에 기록된 WAV와 WWU만 즉시 되돌리려고 시도합니다.

5. Wwise 재시작 후 사후 검증

필수 재시작

headless waapi-server는 외부에서 변경된 WWU를 실행 중에 reload하지 않습니다. 적용 후 Ctrl-C로 중지하고 같은 명령으로 다시 시작하세요.

Post-apply validation
.venv/bin/wwise-p4-source-relocator validate-apply \
  --manifest reports/live-manifest.json \
  --waapi-url ws://127.0.0.1:18080/waapi \
  --report reports/live-validation.md

6. 롤백

Manifest 범위 롤백
.venv/bin/wwise-p4-source-relocator rollback \
  --manifest reports/live-manifest.json

Wwise 복구 상태까지 다시 확인하려면 WAAPI server를 한 번 더 재시작하고 scan 또는 doctor를 실행합니다.

Windows · PowerShell · End-to-end

Windows: Wwise + Perforce 통합 테스트

아래 명령은 저장소 루트에서 PowerShell로 실행합니다. Windows에서는 가상환경 실행 파일과 경로 구분자가 다르며, Wwise fixture 생성 시 --platform Windows를 명시합니다.

필수 구성

Python 3.11+, Wwise Authoring/WwiseConsole, Windows용 p4.exe와 p4d.exe, PowerShell이 필요합니다.

권장 작업 위치

$env:TEMP\wwise-relocator-p4 아래에 server와 workspace를 격리하면 실제 depot 및 기존 Wwise 프로젝트와 분리할 수 있습니다.

1. Python 설치와 읽기 전용 fixture 테스트

가상환경을 activate하지 않고 Python 실행 파일을 직접 호출하므로 PowerShell execution policy를 변경할 필요가 없습니다.

PowerShell · 저장소 루트
py -3.11 --version
py -3.11 -m venv .venv
& .\.venv\Scripts\python.exe -m pip install -e ".[dev,waapi]"
& .\.venv\Scripts\python.exe -m pytest

New-Item -ItemType Directory -Force reports | Out-Null
& .\.venv\Scripts\python.exe -m wwise_p4_source_relocator inspect-wwu `
  --wwu "tests\fixtures\sample_project\Actor-Mixer Hierarchy\Default Work Unit.wwu" `
  --project-root tests\fixtures\sample_project `
  --json-out reports\source-plan.json `
  --markdown-out reports\source-plan.md

py -3.11이 발견되지 않으면 Python 3.11 이상을 설치하거나 동일 버전의 python.exe 전체 경로를 사용합니다. 여기까지는 Wwise와 Perforce 없이 실행할 수 있습니다.

2. Pilot 경로와 WwiseConsole 설정

PowerShell · 환경 변수
$PilotBase = Join-Path $env:TEMP "wwise-relocator-p4"
$PilotWorkspace = Join-Path $PilotBase "workspace"
$PilotRoot = Join-Path $PilotWorkspace "WwiseRelocatorPilot"
$WwiseConsole = Join-Path ${env:ProgramFiles(x86)} "Audiokinetic\Wwise 2025.1.8.9170\Authoring\x64\Release\bin\WwiseConsole.exe"

$env:P4PORT = "127.0.0.1:16661"
$env:P4USER = "pilot"
$env:P4CLIENT = "wwise-relocator-pilot"
$env:PYTHONPATH = "src"
$env:Path = "$(Resolve-Path .\.tools\perforce);$env:Path"

Test-Path $WwiseConsole
p4 -V
p4d -V

Test-Path가 False이면 Wwise Launcher에서 설치 위치를 확인해 $WwiseConsole을 실제 WwiseConsole.exe 경로로 바꾸세요. 버전 폴더는 설치된 Wwise와 일치해야 합니다.

3. 일회용 P4 Server 준비

.tools\perforce에 Windows용 p4.exe와 p4d.exe를 두고 vendor checksum을 검증한 뒤 사용합니다.

PowerShell · 별도 창에서 서버 시작
New-Item -ItemType Directory -Force (Join-Path $PilotBase "server") | Out-Null

& .\.tools\perforce\p4d.exe `
  -r (Join-Path $PilotBase "server") `
  -p $env:P4PORT `
  -L (Join-Path $PilotBase "p4d.log") `
  -J (Join-Path $PilotBase "journal")
Windows에서 P4 Server 2026.1 최초 사용자 bootstrap 보기

첫 실행으로 데이터베이스가 만들어지면 Ctrl-C로 서버를 중지하고 아래 설정을 한 번 적용한 뒤 다시 시작합니다. loopback 전용 test server 외에는 사용하지 마세요.

PowerShell · localhost 전용 설정
$ServerRoot = Join-Path $PilotBase "server"

& .\.tools\perforce\p4d.exe -r $ServerRoot -cset dm.user.noautocreate=0
& .\.tools\perforce\p4d.exe -r $ServerRoot -cset dm.user.setinitialpasswd=1
& .\.tools\perforce\p4d.exe -r $ServerRoot -cset dm.user.resetpassword=0
& .\.tools\perforce\p4d.exe -r $ServerRoot -cset run.users.authorize=0
& .\.tools\perforce\p4d.exe -r $ServerRoot -cset dm.user.hideinvalid=0

설정 후 3번의 server start 명령을 다시 실행하고, 서버가 떠 있는 상태에서 다음 명령으로 최초 사용자와 ticket을 만듭니다.

PowerShell · 서버 재시작 후 인증
& .\.tools\perforce\p4.exe -p $env:P4PORT -u $env:P4USER passwd
& .\.tools\perforce\p4.exe -p $env:P4PORT -u $env:P4USER login

4. Windows P4 client 만들기

PowerShell · client spec
New-Item -ItemType Directory -Force $PilotWorkspace | Out-Null

$ClientSpec = @"

Root: $PilotWorkspace
Options: noallwrite noclobber nocompress unlocked nomodtime normdir
SubmitOptions: submitunchanged
LineEnd: local

$ClientSpec | p4 client -i

macOS의 /tmp canonicalization 문제는 Windows에 없습니다. 대신 Root와 $PilotRoot가 같은 drive의 실제 경로인지, p4 where가 해당 경로를 mapping하는지 확인합니다.

5. Windows용 Wwise fixture 생성

PowerShell · bootstrap-project
& .\.venv\Scripts\python.exe -m wwise_p4_source_relocator bootstrap-project `

Push-Location $PilotWorkspace

Pop-Location

Diagnostics

자주 발생하는 문제

doctor가 .wproj는 찾지만 준비 실패

.wproj가 존재하는 것만으로는 충분하지 않습니다. Originals 아래 WAV, WWU source reference, workspace mapping, WAAPI 연결까지 모두 필요합니다. 비어 있는 프로젝트라면 bootstrap-project로 fixture를 만드세요.

WAAPI 연결 거부 또는 HTTP 403

WAMP port가 명령과 일치하는지 확인하고 --allowed-origin localhost,127.0.0.1를 포함하세요. 일반 개발 서버와 겹치기 쉬운 8080/8090 대신 18080/18090을 권장합니다.

apply 후에도 Wwise가 이전 source path를 반환

외부 WWU 변경이 아직 로드되지 않은 상태입니다. headless WAAPI server를 완전히 중지한 뒤 같은 project와 port로 다시 시작하고 validate-apply를 재실행하세요.

Perforce workspace 바깥 파일로 판정됨

macOS에서 client root가 /tmp/...로 되어 있다면 canonical /private/tmp/...로 바꾸세요. p4 where 결과와 실제 project root가 동일한 workspace 아래인지도 확인합니다.

p4가 성공 코드인데 실제 이동이 실패함

Perforce는 오류 문구를 출력하면서 process exit code 0을 반환하는 경우가 있습니다. 도구는 status output을 검사하지만, 최종 판정 때는 반드시 파일 위치, p4 opened, WWU diff를 함께 확인하세요.

p4 move가 source WAV에서 실패

source WAV가 먼저 edit 상태여야 합니다. 현재 apply 구현은 p4 edit 후 p4 move를 수행합니다. 수동으로 P4 상태를 바꿨다면 rollback으로 정리한 뒤 계획부터 다시 시작하세요.

WwiseConsole이 자동으로 발견되지 않음

macOS에서는 --wwise-console "/absolute/path/to/WwiseConsole.sh", Windows에서는 --wwise-console $WwiseConsole처럼 설치 경로를 명시하세요. Windows의 WwiseConsole.exe는 일반적으로 Wwise 설치 폴더의 Authoring\x64\Release\bin 아래에 있습니다. 생성할 폴더 이름과 --project-name은 맞추는 것이 안전합니다.

PowerShell에서 py -3.11 또는 가상환경 Python을 찾지 못함

py --list로 설치 버전을 확인하세요. Python Launcher가 없다면 Python 3.11 이상의 python.exe 전체 경로로 venv를 만들 수 있습니다. 이후 명령은 activate 없이 .\.venv\Scripts\python.exe를 직접 호출합니다.

Windows에서 p4d.exe가 실행되지 않거나 16661 port가 열리지 않음

Windows용 binary인지 확인하고 Get-FileHash로 vendor checksum을 검증하세요. Windows Defender 또는 조직 보안 정책이 차단했다면 임의 예외를 추가하지 말고 관리자에게 localhost test binary와 port 승인을 요청합니다. Get-NetTCPConnection -LocalPort 16661로 충돌도 확인하세요.

PowerShell 명령의 줄 연속이 끊어짐

backtick(`) 뒤에는 공백이나 주석이 없어야 합니다. 복사한 여러 줄 명령이 실패하면 한 줄로 합치거나 각 backtick이 줄의 마지막 문자인지 확인하세요. 이 문서의 명령은 cmd.exe가 아니라 PowerShell 기준입니다.

Real project

실제 프로젝트에 적용할 때

일회용 pilot에서 동일한 도구 버전으로 apply/validate/rollback을 끝낸 뒤, 실제 Wwise/Perforce 환경에서는 아래 순서를 반복합니다.

  1. 팀의 작업 중지 시간과 대상 Work Unit 담당자를 확인합니다.
  2. 실제 project root에 doctor를 실행하고 보고서를 보존합니다.
  3. scan → plan → validate-plan까지만 실행해 리뷰를 받습니다.
  4. 전용 numbered changelist에 정확히 한 WAV만 apply --only합니다.
  5. Wwise에서 External Project Changes를 reload하거나 Authoring/WAAPI server를 재시작합니다.
  6. validate-apply, Wwise missing source 확인, p4 diff, p4 opened를 함께 검토합니다.
  7. pilot 단계에서는 rollback으로 원복하고 증거를 기록합니다. submit은 별도의 승인 절차입니다.
자동화 중단 조건

shared source, 중복 source, 대상 파일 충돌, 여러 WWU match, GUID/object path 불일치, preflight 오류가 하나라도 있으면 수동 검토로 전환합니다.
